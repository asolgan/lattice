# Follow-up: `projectedFromRevisions` read-then-excluded coverage (12.3 → 12.3-follow-up)

Status: DEFERRED (Winston adjudication, 2026-06-14)

## What 12.3 ships (v1)

`projection.ContributingSources` widens `projectedFromRevisions` (Contract #6 §6.3,
freshness: `auto`) to cover the **contributing-binding** source set: the actor's identity
vertex, the lens-definition vertex, and every roles / tasks / services / links key that
contributed a binding into the actor's winning rows. This is derivable from the same
bindings the Execution path already materializes (the graph keys present in the projected
RETURN rows), so it needs no new instrumentation.

## What is deferred

Sources that were **read then excluded** — e.g. a now-closed task the executor matched and
then dropped via a `WHERE` predicate or the realness filter — are NOT covered. Capturing
them requires **full-executor touched-then-dropped instrumentation**: the `full` executor
reporting every Core-KV key it touched-then-dropped during evaluation. That is a
cross-cutting change to the hot eval path with its own correctness/perf review surface.

## Why deferral is safe

`projectedFromRevisions` is the **coherence/debug provenance** datum, NOT the write-ordering
guard. The ordering guard is `projectionSeq` (Contract #6 §6.2, Story 12.1a), which is
independent and unaffected. Under-covering `projectedFromRevisions` therefore carries **no
authorization risk** — a stale projection still loses to a higher-seq write regardless of
what provenance keys are recorded.

## Trigger to pull this in

A dedicated 12.3-follow-up story that threads a "touched keys" accumulator through the
`full` executor's evaluation. Sized separately from the security-plane 12.3/12.4 work.
