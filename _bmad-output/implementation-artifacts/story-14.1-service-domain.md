## 0. THE HEADLINE — a `service.<x>.instance` is a REAL LINKED VERTEX whose OUTCOME LIVES IN ASPECTS (read first; it governs everything)

This is the one thing to get right. Epic 14 exists to prove the package model carries **real multi-vertex convergence**, not a leaseApp-scalar model (epics ~662–663). 14.1 is the move that **forces** that: a background check / payment is **not** a field on the lease application — it is a **separate `service` vertex, linked** to the applicant identity, whose external-call **outcome** (did the bgcheck pass? when?) is recorded as **aspect(s)** on that vertex. Later (14.4) a convergence lens must fan out **across the `providedTo` link** to read that outcome aspect — which is only possible because 14.1 made the instance a real linked vertex with its outcome in a readable aspect.

So the two load-bearing decisions, both gate-asserted:

1. **The instance is a vertex, linked — not a scalar.** `vtx.service.<id>` with `class: "service.<x>.instance"`, linked `instanceOf` its template and `providedTo` the applicant identity (§2.2). This is what makes 14.4's `MATCH (id)<-[:providedTo]-(inst:service)` possible (§10.2 amendment).
2. **The instance's outcome lives in ASPECTS (D5, invariant b), root `data` minimal — and a TEST/GATE asserts it.** `vtx.service.<id>.<outcomeAspect>` carries `status` + `completedAt`; the vertex root `data` is minimal (just the `class` discriminator + at most an auth scalar — and 14.1 has no auth scalar because the read-path is deferred). The `completedAt` aspect is the **freshness-predicate input** 14.4's convergence lens + 14.5's ADR-51 temporal window read. **§6 requires a test that asserts the outcome is in an aspect and the root `data` is minimal — not just a fixture detail.** (This mirrors how 13.4's harness + 14.5's e2e gate-assert "outcome in aspect, root minimal" rather than leaving it to review.)

Everything else (the template vertex, the lifecycle ops, the op-metas, the permissions, the manifest) is scaffolding that lets these two land cleanly via `InstallPackage`.

### Item B — the `service` vertex-type DDL (the template/instance discriminator)

```
CanonicalName:     "service"
Class:             "meta.ddl.vertexType"
PermittedCommands: [ "CreateServiceTemplate", "CreateServiceInstance", "RecordServiceOutcome" ]   # §2.3 — confirm verb names
```

Vertex shape (Contract #1 §1.1 + D5):

```
vtx.service.<NanoID>                              # the service vertex (template OR instance)
vtx.service.<NanoID>.class                        # aspect: value = "service.<x>.template" | "service.<x>.instance"   (§2.1)
# INSTANCE outcome (D5 — business data in aspect(s), root data minimal):
vtx.service.<NanoID>.outcome                      # aspect: { status, completedAt }   (§2.4 — confirm one-aspect-vs-two)
# LINKS (§2.2):
lnk.service.<tplId>.availableAt.<loc-type>.<locId>     # template availableAt location-ish   (scoped form §3)
lnk.service.<tplId>.providedBy.<prov-type>.<provId>    # template providedBy provider-ish     (scoped form §3)
lnk.service.<instId>.instanceOf.service.<tplId>        # instance instanceOf template
lnk.service.<instId>.providedTo.identity.<applicantId> # instance providedTo identity
```

**Root `data` is minimal (D5, invariant b).** The vertex root carries only the `class` discriminator location decision: **prefer the `class` on a `.class` aspect** (mirroring identity's `state` aspect and orchestration-base's task whose discriminating scalars live on root only because the cap-lens reads them — here NOTHING reads `service` on the read-path, §3, so there is no cap-lens-read justification to put anything on root). The minimal-root rule is the default; the read-path exception does not apply. **Confirm the placement of `class`** in §6's D5 test (Open Question Q1): the safest D5-faithful shape is root `data: {}` (or `data: {"class":"service.<x>.instance"}` ONLY if you judge the class discriminator an auth/structural scalar) + the outcome strictly in the `.outcome` aspect. The **load-bearing** D5 assertion (§6) is that the **outcome** (`status`/`completedAt`) is in an **aspect**, never on root.

Self-description aspects (the installer **rejects** a DDL missing any — `build.go` ~74–88): `InputSchema` (the union of all three ops' payload fields), `OutputSchema` (`{primaryKey}`), `FieldDescription` (every payload field), `Examples` (at least one per op — mirror `orchestration-base`'s `CreateTask` example richness). `Description` describes the vertex shape + the template/instance distinction + the link directions (run the sentence test in the prose) + the D5 placement, present-tense, no history.

### Item C — the Starlark script (mirror identity-domain's aspect-writing + orchestration-base's link-writing)

The script's `execute(state, op)` branches on `op.operationType`. Reuse the proven helpers from `orchestration-base/ddls.go` (`make_vtx`, `make_link`, `split_key`, `required_string`, `parts_of`, `vertex_alive`, `bare_nanoid_or_mint`) and identity-domain's aspect-create shape (`{"op":"create","key": key + ".<aspect>","document":{"class":"<aspect>","vertexKey": key,"localName":"<aspect>","isDeleted":False,"data":{…}}}`, `ddls.go` ~193–220). Key points:

- **`CreateServiceTemplate`** — mint (or accept a caller-supplied bare-NanoID id) `vtx.service.<id>`; write the `.class` aspect = `"service.<x>.template"`; create the `availableAt` + `providedBy` links to the (pre-existing, validated-by-known-key) endpoints the caller names; emit a `service.templateCreated` event; return `{primaryKey: tpl_key}`. **Validate both link endpoints exist** (no-orphan, FR29/P4 — mirror orchestration-base's `vertex_alive` checks on every link endpoint). Root `data` minimal.
- **`CreateServiceInstance`** — mint `vtx.service.<id>`; write `.class` = `"service.<x>.instance"`; create the `instanceOf` link to the named template (validated alive) and the `providedTo` link to the named applicant **identity** (validated alive — this is the cross-vertex link that forces 14.4's convergence). **Root `data` minimal — NO outcome yet** (the instance is created in a "pending" state with the outcome aspect either absent or `{status:"pending", completedAt:null}` — confirm in Q2). Emit `service.instanceCreated`. Return `{primaryKey: inst_key}`. **The caller-supplied-bare-NanoID seam** (`taskId`-style, orchestration-base ~123–139) should be supported on the instance create so that **14.4's `instanceOp` can supply Loom's write-ahead bare handle** as the instance id (§10.6: the instanceOp DDL "prepends its package-chosen type to form `vtx.<type>.<handle>`" — that package-chosen type is `service`, and the handle is the caller-supplied id). Build the seam now (it is cheap and forward-fits 14.4); name it e.g. `instanceId`. CreateOnly semantics make a crash-retry with the same id collapse on the Contract #4 tracker.
- **`RecordServiceOutcome`** — the op that records the external-call result. GET the instance (alive, is an `instance` not a `template`); write the **outcome aspect** `vtx.service.<id>.outcome` (or `.status` + `.completedAt`; Q2) with `{status: <terminal>, completedAt: <RFC3339-UTC>}`; **OCC-guard** the write (assert the read revision if updating, Contract #2 §2.6 — mirror orchestration-base's `transition_task` `expectedRevision`); emit `service.outcomeRecorded`. **Normalize `completedAt` to canonical UTC whole-second RFC3339** via `time.rfc3339_utc(...)` (exactly as orchestration-base does for `expiresAt`, `ddls.go` ~174) so 14.4's lexical freshness compare (`completedAt + window > now`) is sound regardless of caller formatting. **This op is the shape 14.4's `replyOp` DDL reuses** — 14.4's replyOp will do this same outcome-aspect write, additionally emitting `orchestration.externalTaskCompleted` with `payload.externalRef = <instanceKey>` (§10.6). 14.1 builds the plain outcome-record; do NOT add the externalTaskCompleted/externalRef machinery (14.4).

**`status` enum** — define the terminal set the demo needs (recommend `passed` / `failed` for bgcheck; `succeeded` / `declined` for payment — or a unified `completed`/`failed` if you keep one `<x>` family). Confirm in Q3. The freshness predicate (14.4/14.5) reads `completedAt`; the `violating` decision reads `status`. Keep the enum small and demo-scoped.

## 2.2. The link sentence test (run it on every link — Contract #1 §1.1)

Contract #1 §1.1: "source <relation> target", the **later-arriving vertex is the SOURCE**. Run the sentence test on each:

| Link | Key shape | Source → Target | Sentence | Verdict |
|---|---|---|---|---|
| `availableAt` | `lnk.service.<tplId>.availableAt.<locType>.<locId>` | template → location | "**service** (template) **availableAt** **location**" | ✓ reads as a sentence; template is the later-arriving source, the location pre-exists (matches `CONCEPT.md` §6.9 + the bootstrap remnant) |

## 2.4. D5 — the outcome in aspect(s), root minimal (AC #2, invariant b — the gate-asserted decision)
