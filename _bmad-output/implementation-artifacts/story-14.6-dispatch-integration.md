## Why this story exists

Story 14.5 (the first all-engines e2e) proved that the orchestration **dispatch** plane has never been exercised through the **real** Processor: every Loom/Weaver per-engine test uses a `fakeProcessor` keyed by `operationType`, while the real Processor (`internal/processor/step4_hydrate.go resolveClass`) resolves a DDL by **`class`** and hydrates OCC reads from **`ContextHint.Reads`** — neither of which the engines set on dispatch. So in production every op Weaver/Loom dispatch is rejected (`NoDDLForClass`/`UnknownApplicant`). 14.5's e2e only converges via a documented harness `normalizeDispatch` shim. All three 14.5 review layers confirmed these as real, product-level gaps. See memory `project_dispatch_plane_gap`. 14.5's `applyMatch` engine bugfix already landed standalone (e560188); 14.5's vertical (freshUntil lens + harness + gate) stays uncommitted until this story lets the harness drop the shim.

## Design

### RF#1 — Processor operationType→class reverse index (the contractually-anticipated fix)

Contract #2 §2.1 already says `class` is transitional, "to be removed once the DDL cache fully covers operationType→class derivation." That index was never built. Build it:

- `internal/processor/ddl_cache.go`: add `byCommand map[string]string` (operationType → canonicalName). Populate during `Refresh`/`Invalidate` by iterating `byName` and, for each ref, mapping every `PermittedCommands` entry → that ref's `CanonicalName`. **Ambiguity guard:** if two DDLs admit the same operationType, do NOT index it (leave it requiring explicit `class`) and log a warning — fail-closed, never guess.
- `internal/processor/step4_hydrate.go resolveClass`: after the `env.Class` and `payload.class` checks, fall back to `cache.ClassForCommand(env.OperationType)`. Miss/ambiguous → the existing "must carry class" error (unchanged behavior for genuinely-unresolvable ops).
- **Security note:** this does NOT weaken auth — the inferred class is exactly the DDL that admits the operationType (the same `PermittedCommands` the permission check already keys on), so there is no new trust surface; it eliminates class/operationType mismatch. Gate 2/Gate 3 must stay green.
- **Contract #2 §2.1:** amend in-place (UNCOMMITTED) — the index now exists; `class` is optional and resolved from operationType when omitted; clients/engines may omit it.

### RF#2 — engine-dispatched ops declare their reads (RECOMMENDED: the dispatcher declares what it touches)

The engines already KNOW the vertex keys their op operates on (they build the payload). So the dispatcher sets `ContextHint.Reads` from those keys + their `.state` aspects:

- **Loom** (`actuator.go`/`engine.go`): add a `Reads []string` to `outboxRecord` + the dispatched `opEnvelope` (it already has the field shape minus reads). For the externalTask `instanceOp`, set `Reads = [subjectKey, subjectKey+".state"]` (the instanceOp's `vertex_alive(subjectKey)` contract). For systemOp steps that read, same pattern from the step's known target.
- **Weaver** (`actuator.go` `submit` + `strategist.go`): add `ContextHint.Reads` to the Weaver `opEnvelope` and have the strategist's `plan` carry the read-set. For `CreateTask`: `Reads = [assignee, assignee.state, forOperation, forOperation.state, scopedTo, scopedTo.state]` (the three `vertex_alive` keys its DDL documents). For `StartLoomPattern`: whatever its DDL reads (verify — it is "event-only", likely just the subject; confirm during dev).
- **Open fork for team-review:** dispatcher-declares (above) vs. a Processor default-read-set derived from the op's target/subject + payload vertex-valued fields. Recommend dispatcher-declares (explicit, no Processor-side guessing, matches the existing client contract where the caller declares reads). The team-review should pressure-test whether a per-op read-set belongs in the engine or should be DDL-declared metadata.

### RF#3 — the `MarkExpired` DDL (hardest part; polymorphic entity)

`MarkExpired{entityKey, targetId, expiredAt}` fires for an entity of ANY anchor type (here `leaseapp`). Its job: re-touch the entity so Refractor reprojects its convergence row, which (under 14.4's compute-from-`validUntil` model) re-evaluates `validUntil > $now` (now false) → `missing_bgcheck` re-opens. It writes nothing the lens reads — it just needs to trigger a CDC reprojection of the anchor.

- **Open fork for team-review (the polymorphism wrinkle):**
  - **(a) Type-agnostic touch op** — a `MarkExpired` DDL that writes a generic `freshnessExpiry` marker aspect on `entityKey` (`vtx.<type>.<id>.freshnessExpiry = {expiredAt}`), resolving the entity's own class from its existing `.class` rather than the op's class. Needs the document-less/`class==""` path (or a dedicated platform class) so a single DDL serves all anchor types. Cleanest IF the engine + lens reproject on an anchor-aspect write the lens doesn't read.
  - **(b) Per-target DDL** — each weaver-target package declares its own `MarkExpired`-equivalent for its anchor type (lease-signing ships `leaseapp`'s). More boilerplate; less generic; but class-keyed cleanly.
  - **Recommendation: (a)**, with a hard check during dev that a `freshnessExpiry`-aspect write on the anchor actually triggers the actorAggregate reprojection (it should — the lens anchors on the entity). If reprojection does NOT trigger on an unread anchor-aspect, fall back to bumping a column the lens DOES read, or (b).
- **Where it lives:** `MarkExpired` is generic platform vocabulary → orchestration-base (not lease-signing), per the "generic substrate" placement. Confirm in team-review.
- Determinism: the temporal lane already derives a §10.4 requestId (schedule subject + fire instant) → Contract #4 tracker collapses redelivery. The DDL must be safe under at-least-once (idempotent re-touch).
